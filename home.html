Basics of REST

REST (REpresentational State Transfer) is a simple stateless architecture that generally runs over HTTP.REST is often used in mobile applications, social networking Web sites, mashup tools and automated business processes. The REST style emphasizes that interactions between clients and services is enhanced by having a limited number of operations (verbs). Flexibility is provided by assigning resources (nouns) their own unique universal resource indicators (URIs). Because each verb has a specific meaning (GET, POST, PUT and DELETE), REST avoids ambiguity.

It is resource based unlike action based. When I say resource based I mean that various endpoints of an api correspond to some resources which could be a database or an outcome of a computing algorithm. Where as action based means telling the machine what to do like GET or Post. So with rest we get the resource information rather doing actions unlike soap.

The resources are manipulated to be represented in any way which can be used in the desirable form. It can be represented as whole or a part of itself. Can be represented as json or xml. For example, resource = person(Ravi), Service= contact information(GET), representaion=name, address, phone number, json/xml.

Following are the six constraints required for an API to qualify as REST:

  • Uniform Interface

    It defines the interface between a client and a server. In broad sense it means that REST gets identified by http verbs(PUT, GET, POST, DELETE), resource(uri) and response(status and body) which simplifies and decouples the architecture which in turn enables each part of the application to evolve independently. The four guiding principles of Uniform Interfaces are:

    • Resource Based

      As explained above.

    • Manipulaltion of Resources through Representation

      As explained above.

    • Self descriptive Messages

      Each message is packed with enough information, which describes to process itself.

    • Hypermedia as the Engine of Application State (HATEOAS)

      Services deliver state to clients via body content, response codes, and response headers. This is technically referred-to as hypermedia (or hyperlinks within hypertext).

  • Stateless

    It means that state which is required to handle the request is contained within the request itself, either as part of the URI, query-string parameters, body, or headers. This allows servers not to take part in the state and session maintenance at client's end.

  • Cacheable

    As on the World Wide Web, clients can cache responses. Responses must therefore, implicitly or explicitly, define themselves as cacheable, or not, to prevent clients reusing stale or inappropriate data in response to further requests. Well-managed caching partially or completely eliminates some clientâ€“server interactions, further improving scalability and performance.

  • Client Server

    Rest assumes interaction between disconnected components. Uniform interface constraint links both of them.

  • Layered System

    It represents both client-server and cacheability constraints. There could be software or hardware intermediaries between client and server. And the client could take the data from the cache before it expires in it before refreshing the same from the darabase.

  • Code on Demand

    Server can send logic to the client in form of java applets or javascript. and this is option constraint, which means that an api can qualify as REST without complying with this particular constraint.

Abiding to the REST architecture by designing the API in accordance to the above mentioned constraint would ensure any distributed application infrastructure to have all desirable properties like performance, scalability, simplicity, modifiability, visibility, portability and reliability.

REST or Web-Service, which one to choose? You Decide.

Rest

  • The RESTful Web services are completely stateless. This can be tested by restarting the server and checking if the interactions are able to survive.
  • Restful services provide a good caching infrastructure over HTTP GET method (for most servers). This can improve the performance, if the data the Web service returns is not altered frequently and not dynamic in nature.
  • The service producer and service consumer need to have a common understanding of the context as well as the content being passed along as there is no standard set of rules to describe the REST Web services interface.
  • It is particularly useful for restricted-profile devices such as mobile and PDAs for which the overhead of additional parameters like headers and other SOAP elements are less.
  • It services are easy to integrate with the existing websites and are exposed with XML so the HTML pages can consume the same with ease. There is hardly any need to refactor the existing website architecture. This makes developers more productive and comfortable as they will not have to rewrite everything from scratch and just need to add on the existing functionality.
  • It-based implementation is simple compared to SOAP.

Webservice

  • The Web Services Description Language (WSDL) contains and describes the common set of rules to define the messages, bindings, operations and location of the Web service. WSDL is a sort of formal contract to define the interface that the Web service offers.
  • It requires less plumbing code than REST services design, (i.e., transactions, security, coordination, addressing, trust, etc.) Most real-world applications are not simple and support complex operations, which require conversational state and contextual information to be maintained. With the SOAP approach, developers need not worry about writing this plumbing code into the application layer themselves.
  • Web services (such as JAX-WS) are useful in handling asynchronous processing and invocation.
  • It supports several protocols and technologies, including WSDL, XSDs, SOAP, WS-Addressing
